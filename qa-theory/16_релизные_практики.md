# Релизные практики

Стратегии развёртывания, которые минимизируют риск при выкатке новой версии.

## Blue-Green Deployment

Два идентичных окружения: Blue (текущее) и Green (новое).

```
              ┌──────────────┐
Пользователи ─┤  Балансировщик │
              └──┬───────┬───┘
                 │       │
            ┌────▼──┐ ┌──▼────┐
            │ Blue  │ │ Green │
            │ v1.0  │ │ v2.0  │
            │(active)│ │(idle) │
            └───────┘ └───────┘
            
1. Деплоим v2.0 на Green
2. Тестируем Green
3. Переключаем балансировщик на Green
4. Green становится active, Blue — idle
5. Если баг → мгновенный откат: переключаем обратно на Blue
```

| Плюсы | Минусы |
|-------|--------|
| Мгновенный откат | Двойные ресурсы (два окружения) |
| Нулевой простой | Сложнее с БД (миграции должны быть совместимы) |
| Можно тестировать перед переключением | Дороже в инфраструктуре |

## Canary Deployment (Канареечный деплой)

Новая версия выкатывается на **небольшой процент** пользователей. Если всё ок — постепенно увеличиваем.

```
Шаг 1:  5% трафика  → v2.0    95% → v1.0
Шаг 2:  25% трафика → v2.0    75% → v1.0
Шаг 3:  50% трафика → v2.0    50% → v1.0
Шаг 4:  100% трафика → v2.0   (v1.0 выводится)
```

| Плюсы | Минусы |
|-------|--------|
| Минимальный риск | Сложнее настроить маршрутизацию |
| Реальные пользователи тестируют | Две версии работают одновременно |
| Можно откатить на любом этапе | Нужен мониторинг метрик |

**Название:** Шахтёры брали канареек в шахту — если птица умирала, значит, воздух отравлен.

## Feature Toggles (Feature Flags)

Новая функциональность скрыта за «переключателем» в коде. Можно включать/выключать без деплоя.

```javascript
if (featureFlags.isEnabled('new_checkout')) {
  showNewCheckout();
} else {
  showOldCheckout();
}
```

### Виды флагов

| Тип | Назначение | Пример |
|-----|-----------|--------|
| **Release toggle** | Скрыть незаконченную фичу | Новый дизайн корзины |
| **Experiment toggle** | A/B тестирование | Два варианта кнопки |
| **Ops toggle** | Включить/выключить под нагрузкой | Отключить тяжёлые рекомендации |
| **Permission toggle** | Доступ для конкретных пользователей | Beta-функции для тестировщиков |

| Плюсы | Минусы |
|-------|--------|
| Деплой ≠ релиз (разделение) | Технический долг (флаги нужно убирать) |
| Мгновенное включение/выключение | Комбинаторная сложность (много флагов) |
| A/B тестирование | Нужно тестировать оба состояния |

**Инструменты:** LaunchDarkly, Unleash, Firebase Remote Config, самописные.

## Rolling Deployment

Обновление серверов **по одному** (или группами). В любой момент часть серверов на старой версии, часть — на новой.

```
Сервер 1: v1.0 → v2.0 ✓
Сервер 2: v1.0 → v2.0 ✓  (пока обновляется, трафик на 1 и 3)
Сервер 3: v1.0 → v2.0 ✓
```

| Плюсы | Минусы |
|-------|--------|
| Не нужны доп. ресурсы | Две версии одновременно (совместимость) |
| Нулевой простой | Медленный откат |

## Сравнение стратегий

| Стратегия | Простой | Скорость отката | Ресурсы | Риск |
|-----------|:---:|:---:|:---:|:---:|
| **Blue-Green** | 0 | Мгновенно | 2x | Минимальный |
| **Canary** | 0 | Быстро | 1x + % | Минимальный |
| **Feature Toggles** | 0 | Мгновенно | 1x | Низкий |
| **Rolling** | 0 | Медленно | 1x | Средний |
| **Recreate** (всё остановить, обновить, запустить) | Есть | Медленно | 1x | Высокий |

---

## Для тестировщика

- Понимать, какая стратегия используется в проекте
- При canary — тестировать на канареечном окружении
- При feature flags — тестировать **оба состояния** флага (вкл/выкл)
- Проверять совместимость API между версиями (v1 и v2 работают одновременно)
- Знать, как сделать откат и что при этом происходит с данными
