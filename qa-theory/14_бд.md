# Базы данных

**База данных** — организованное, структурированное хранилище электронных данных. Управление осуществляется с помощью СУБД (систем управления базами данных).

## Типы БД

### Реляционные (SQL)

Данные хранятся в связанных таблицах (строки и столбцы) с предопределённой схемой.

- Данные нормализованы и связаны внешними ключами
- Масштабируются вертикально (увеличение мощности сервера)
- Обеспечивают строгую согласованность (ACID)
- Примеры: PostgreSQL, MySQL, Oracle Database, Microsoft SQL Server

### Нереляционные (NoSQL)

Используются для неструктурированных данных.

- Данные часто денормализованы для быстрого чтения
- Масштабируются горизонтально (добавление серверов)
- Часто жертвуют строгой согласованностью ради производительности (BASE)
- Примеры: MongoDB (документная), Redis (ключ-значение), Cassandra (колоночная), Neo4j (графовая)

### Что выбрать?

| SQL | NoSQL |
|-----|-------|
| Данные структурированы | Данные неструктурированы |
| Высокая надёжность транзакций | Быстрая разработка |
| Сложная аналитика | Горизонтальное масштабирование |
| Банки, ERP, CRM | Соцсети, IoT, каталоги товаров |

## JOIN-ы

| Тип | Описание |
|-----|----------|
| **INNER JOIN** | Только совпадения из обеих таблиц |
| **LEFT JOIN** | Все из левой таблицы + совпадения из правой |
| **RIGHT JOIN** | Все из правой таблицы + совпадения из левой |
| **FULL JOIN** | Все из обеих таблиц |

`WHERE` фильтрует **после** JOIN, а условие в `ON` фильтрует **во время** JOIN. Для INNER JOIN разницы нет, а для LEFT/RIGHT/FULL — принципиальная.

```sql
-- INNER: только совпадения
SELECT * FROM A INNER JOIN B ON A.id = B.a_id;

-- LEFT: все из A, + совпадения из B
SELECT * FROM A LEFT JOIN B ON A.id = B.a_id;
```

## Нормализация vs Денормализация

**Нормализация** — организация данных для уменьшения избыточности.

Нормальные формы:
1. **1NF** — атомарность значений (нет повторяющихся групп)
2. **2NF** — 1NF + нет частичных зависимостей от составного ключа
3. **3NF** — 2NF + нет транзитивных зависимостей (неключевые атрибуты зависят только от ключа)

**Денормализация** — намеренное нарушение нормальных форм для повышения производительности (чтение важнее записи).

| Нормализация | Денормализация |
|-------------|---------------|
| Меньше дублирования | Быстрее чтение |
| Сложные JOIN | Простые запросы |
| OLTP (транзакции) | OLAP (аналитика) |

## Связи в реляционных БД

- **1:1** — один к одному (редко, разделение таблиц по безопасности/размеру)
- **1:N** — один ко многим (основная)
- **N:M** — многие ко многим (через промежуточную таблицу-связку)

## Первичный ключ (PK)

Уникально идентифицирует запись. Свойства: уникальность, не NULL, неизменяемость, минимальность.

## Транзакция и ACID

**Транзакция** — логическая единица работы (набор операций).

- **A**tomicity — атомарность (всё или ничего)
- **C**onsistency — согласованность (переход из одного валидного состояния в другое)
- **I**solation — изоляция (параллельные транзакции не мешают друг другу)
- **D**urability — долговечность (после commit данные сохранены)

## Индексы

Ускорение поиска: O(log n) вместо O(n).

| Тип | Применение |
|-----|-----------|
| **B-tree** | Стандарт: =, >, <, LIKE 'text%' |
| **Hash** | Только точное совпадение (=) |
| **GiST/GIN** | Полнотекстовый поиск, JSON |

Минусы: замедляют INSERT/UPDATE, занимают место.

```sql
CREATE INDEX idx_name ON users(last_name, first_name);
-- Работает для: last_name
-- Работает для: last_name + first_name
-- НЕ работает для: first_name (отдельно)
```

## Грязное чтение (Dirty Read)

Чтение незафиксированных (uncommitted) данных другой транзакции. Предотвращается уровнем изоляции READ COMMITTED и выше.

## Оптимизация БД

- Индексы (B-tree, Hash, GiST)
- Нормализация/денормализация
- Партиционирование (разделение большой таблицы на части)
- Шардирование (горизонтальное разделение по серверам)
- Оптимизация запросов (EXPLAIN ANALYZE)
- Кэширование (Redis)
- Материализованные представления

**Репликация** — копирование данных на другие серверы (для чтения и отказоустойчивости).
**Шардирование** — горизонтальное разделение данных по разным серверам (для масштабирования записи).
