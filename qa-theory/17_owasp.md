# Уязвимости веб-приложений (OWASP Top 10)

**OWASP** (Open Web Application Security Project) — некоммерческая организация, составляющая рейтинг самых распространённых уязвимостей.

## OWASP Top 10 (2021)

### 1. Broken Access Control (Нарушение контроля доступа)

Пользователь получает доступ к данным или функциям, которые ему не положены.

**Примеры:**
- Изменив `user_id` в URL (`/orders/123` → `/orders/124`), можно увидеть чужой заказ (IDOR)
- Обычный пользователь может вызвать админские API-эндпоинты
- Доступ к файлам других пользователей по прямой ссылке

**Что проверять:**
- Попробовать подставить чужой ID в запрос
- Обратиться к админским эндпоинтам с обычным токеном
- Проверить вертикальную (роли) и горизонтальную (другие пользователи) авторизацию

### 2. Cryptographic Failures (Криптографические ошибки)

Чувствительные данные передаются или хранятся без шифрования.

**Примеры:**
- Пароли хранятся в открытом виде или MD5
- Данные передаются по HTTP (без TLS)
- API возвращает номера карт целиком в ответе

**Что проверять:**
- Все формы логина/регистрации работают по HTTPS
- Пароли не отображаются в логах, API-ответах, URL
- Сессионные куки имеют флаг `Secure`

### 3. Injection (Инъекции)

Недоверенные данные отправляются в интерпретатор как часть команды.

**SQL-инъекция:**
```
Ввод в поле логина: ' OR 1=1 --
Запрос становится: SELECT * FROM users WHERE email='' OR 1=1 --'
Результат: доступ ко всем пользователям
```

**XSS (Cross-Site Scripting):**
```
Ввод в поле имени: <script>alert(document.cookie)</script>
Если не экранировано → скрипт выполнится у других пользователей
```

**Что проверять:**
- Вводить `' OR 1=1 --` в поля ввода
- Вводить `<script>alert(1)</script>` и проверять, выполняется ли
- Сервер не должен возвращать 500 на спецсимволы

### 4. Insecure Design (Небезопасный дизайн)

Архитектурные ошибки, которые нельзя исправить патчем.

**Примеры:**
- Нет лимита попыток ввода пароля → брутфорс
- Восстановление пароля через «секретный вопрос» (легко угадать)
- Нет rate limiting на API

### 5. Security Misconfiguration (Неверная конфигурация)

**Примеры:**
- Включённые дебаг-страницы на проде (stack trace в ответе)
- Дефолтные пароли на сервисах (admin/admin)
- Открытые порты, ненужные сервисы
- CORS: `Access-Control-Allow-Origin: *` с `Credentials: true`
- Серверные заголовки раскрывают версию ПО (`Server: Apache/2.4.51`)

### 6. Vulnerable Components (Уязвимые компоненты)

Использование библиотек с известными уязвимостями.

**Что проверять:**
- `npm audit` — уязвимости в Node.js-зависимостях
- Версии библиотек в `package-lock.json`
- Snyk, Dependabot — автоматический мониторинг

### 7. Authentication Failures (Ошибки аутентификации)

**Примеры:**
- Нет защиты от брутфорса (без lockout/rate limit)
- Слабая политика паролей
- Токены не инвалидируются при логауте
- Session ID в URL (виден в логах, истории)

### 8. Data Integrity Failures (Нарушение целостности данных)

Код или данные из ненадёжного источника без проверки.

**Пример:** Десериализация ненадёжных данных, подмена зависимостей (supply chain attack).

### 9. Logging & Monitoring Failures (Недостатки логирования)

Атаки не обнаруживаются, потому что нет логов или мониторинга.

**Что проверять:**
- Неудачные попытки входа логируются
- Логи не содержат паролей и токенов
- Есть алерты на аномальную активность

### 10. SSRF (Server-Side Request Forgery)

Сервер делает запрос по URL, который контролирует атакующий.

**Пример:**
```
POST /api/fetch-url
{ "url": "http://169.254.169.254/latest/meta-data/" }
→ Сервер обращается к внутреннему сервису AWS и возвращает секреты
```

## Виды XSS

| Тип | Описание | Пример |
|-----|----------|--------|
| **Stored (Хранимый)** | Скрипт сохраняется в БД, выполняется у всех | Комментарий со `<script>` |
| **Reflected (Отражённый)** | Скрипт в URL, выполняется при переходе по ссылке | `?search=<script>alert(1)</script>` |
| **DOM-based** | Скрипт выполняется на стороне клиента через DOM | `document.innerHTML = userInput` |

## Быстрый чек-лист безопасности для QA

- [ ] SQL-инъекция: `' OR 1=1 --` в полях ввода
- [ ] XSS: `<script>alert(1)</script>` в полях и URL
- [ ] IDOR: подмена ID в URL/запросе
- [ ] Брутфорс: нет блокировки после N попыток
- [ ] HTTPS: все страницы и API по HTTPS
- [ ] Куки: `HttpOnly`, `Secure`, `SameSite`
- [ ] Ошибки не раскрывают stack trace
- [ ] Нет дефолтных паролей
- [ ] Логаут инвалидирует токен/сессию
