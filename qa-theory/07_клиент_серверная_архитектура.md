# Клиент-серверная архитектура и HTTP

## Клиент-серверная архитектура

Модель организации вычислительных систем, в которой задачи распределены между клиентами и серверами. Клиент запрашивает услуги или ресурсы у сервера, который отвечает на запросы.

```
       Клиент-серверная архитектура

  1 звено              2 звено
  ┌────────┐  запрос  ┌────────┐     ┌──────┐
  │ Клиент │────────→│ Сервер │────→│  БД  │
  │        │←────────│        │←────│      │
  └────────┘  ответ  └────────┘     └──────┘
```

### Компоненты

1. **Клиент** — программа, с которой пользователь работает напрямую (браузер, мобильное приложение)
2. **Сервер** — компьютер, принимающий запросы, обрабатывающий их и отправляющий ответы
3. **База данных (БД)** — хранилище, где лежит вся информация: пользователи, товары, заказы

Современные клиент-серверные системы часто включают промежуточные элементы: балансировщики нагрузки, прокси-серверы, CDN.

## Структура HTTP-запроса

```
POST /api/orders HTTP/1.1              ← Стартовая строка (метод, путь, версия)
Host: api.shop.example.com             ← Заголовки
Content-Type: application/json
Authorization: Bearer eyJhbG...
Content-Length: 52
                                        ← Пустая строка (разделитель)
{"items": [{"product_id": 42}]}        ← Тело запроса (body)
```

## Структура HTTP-ответа

```
HTTP/1.1 201 Created                   ← Стартовая строка (версия, статус, описание)
Content-Type: application/json          ← Заголовки
X-Request-Id: abc-123
Set-Cookie: session=xyz; HttpOnly
                                        ← Пустая строка
{"id": 789, "status": "new"}           ← Тело ответа
```

## HTTP-методы

| Метод | Действие | Идемпотентный | Тело запроса |
|-------|----------|:---:|:---:|
| **GET** | Получить данные | Да | Нет |
| **POST** | Создать ресурс | Нет | Да |
| **PUT** | Полное обновление | Да | Да |
| **PATCH** | Частичное обновление | Нет* | Да |
| **DELETE** | Удалить ресурс | Да | Нет |

## Коды ответов HTTP

| Группа | Диапазон | Значение | Примеры |
|--------|----------|----------|---------|
| **1xx** | 100-199 | Информационные | 101 Switching Protocols (WebSocket) |
| **2xx** | 200-299 | Успех | 200 OK, 201 Created, 204 No Content |
| **3xx** | 300-399 | Перенаправление | 301 Moved Permanently, 302 Found, 304 Not Modified |
| **4xx** | 400-499 | Ошибка клиента | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 422 Unprocessable Entity, 429 Too Many Requests |
| **5xx** | 500-599 | Ошибка сервера | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout |

### Часто путают

| Код | Когда |
|-----|-------|
| **401** | Не аутентифицирован (нет токена или невалидный) |
| **403** | Аутентифицирован, но нет прав (user → админский эндпоинт) |
| **400** | Некорректный формат запроса (невалидный JSON) |
| **422** | Формат ок, но данные не прошли валидацию (email без @) |

## Основные заголовки

### Заголовки запроса

| Заголовок | Назначение | Пример |
|-----------|-----------|--------|
| `Host` | Домен сервера | `api.shop.com` |
| `Authorization` | Токен / учётные данные | `Bearer eyJ...` |
| `Content-Type` | Формат тела запроса | `application/json` |
| `Accept` | Какой формат ответа хочет клиент | `application/json` |
| `User-Agent` | Информация о клиенте | `Mozilla/5.0...` |
| `Cookie` | Куки | `session_id=abc123` |

### Заголовки ответа

| Заголовок | Назначение |
|-----------|-----------|
| `Content-Type` | Формат тела ответа |
| `Set-Cookie` | Установить куку |
| `Cache-Control` | Управление кэшированием |
| `Access-Control-Allow-Origin` | CORS |
| `Location` | Куда перенаправить (3xx) |

## REST API — соглашения

| Действие | Метод | URL | Ответ |
|----------|-------|-----|-------|
| Список | GET | `/products` | 200 + массив |
| Один ресурс | GET | `/products/42` | 200 + объект |
| Создать | POST | `/products` | 201 + созданный объект |
| Обновить полностью | PUT | `/products/42` | 200 + обновлённый |
| Обновить частично | PATCH | `/products/42` | 200 + обновлённый |
| Удалить | DELETE | `/products/42` | 204 No Content |

## Монолит vs Микросервисы

| | Монолит | Микросервисы |
|---|---|---|
| Структура | Одно приложение | Много маленьких сервисов |
| Деплой | Всё сразу | Каждый сервис отдельно |
| Масштабирование | Всё приложение | Только нужный сервис |
| Отказоустойчивость | Упал = упало всё | Упал один — остальные работают |
