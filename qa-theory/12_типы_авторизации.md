# Типы авторизации и аутентификации

## Аутентификация vs Авторизация

| | Аутентификация (AuthN) | Авторизация (AuthZ) |
|---|---|---|
| Вопрос | **Кто ты?** | **Что тебе можно?** |
| Когда | Первый шаг (логин) | После аутентификации |
| Пример | Ввод логина и пароля | Проверка роли (admin, user) |
| Ошибка HTTP | 401 Unauthorized | 403 Forbidden |

---

## Basic Auth

Логин и пароль передаются в заголовке каждого запроса, закодированные в Base64.

```
Authorization: Basic dXNlcjpwYXNzd29yZA==
                      └── base64("user:password")
```

| Плюсы | Минусы |
|-------|--------|
| Простейшая реализация | Пароль в каждом запросе (нужен HTTPS!) |
| Поддерживается везде | Нет механизма логаута |
| | Base64 ≠ шифрование (легко декодировать) |

**Применение:** внутренние API, Jenkins, простые сервисы.

## Session-Based (Сессионная аутентификация)

Сервер создаёт сессию, хранит её у себя, отправляет клиенту Session ID в куке.

```
1. POST /login { email, password }
2. Сервер: создаёт сессию в памяти/БД, генерирует session_id
3. Set-Cookie: session_id=abc123; HttpOnly; Secure
4. Последующие запросы: Cookie: session_id=abc123
5. Сервер: ищет сессию по session_id → находит пользователя
```

| Плюсы | Минусы |
|-------|--------|
| Простая реализация | Состояние хранится на сервере (stateful) |
| Легко инвалидировать (удалить сессию) | Плохо масштабируется (нужен shared storage) |
| HttpOnly-кука защищена от XSS | Уязвима к CSRF без `SameSite` |

## JWT (JSON Web Token)

Сервер генерирует подписанный токен, клиент хранит его и отправляет в заголовке.

### Структура JWT

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjoxMjN9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
└──── Header ─────────┘└──── Payload ──────────┘└──────── Signature ─────────────────────────┘
```

- **Header:** алгоритм подписи (HS256, RS256)
- **Payload:** данные (user_id, role, exp — время жизни)
- **Signature:** подпись = HMAC(header + payload, secret)

### Как работает

```
1. POST /login { email, password }
2. Сервер: проверяет, генерирует JWT (access_token + refresh_token)
3. Клиент: хранит access_token в памяти / localStorage
4. Последующие запросы: Authorization: Bearer eyJhbG...
5. Сервер: проверяет подпись токена (не обращается к БД!)
```

### Access Token + Refresh Token

| | Access Token | Refresh Token |
|---|---|---|
| Срок жизни | Короткий (15 мин — 1 час) | Длинный (7-30 дней) |
| Где хранить | В памяти / localStorage | HttpOnly cookie |
| Для чего | Доступ к API | Получение нового access token |
| При компрометации | Малый ущерб (быстро истекает) | Нужно инвалидировать |

| Плюсы JWT | Минусы JWT |
|-----------|-----------|
| Stateless (не нужна БД сессий) | Нельзя инвалидировать до истечения |
| Хорошо масштабируется | Размер больше, чем session_id |
| Работает между сервисами (микросервисы) | При утечке — доступ до expiry |

## OAuth 2.0

Протокол **делегированной авторизации**. Позволяет приложению получить доступ к ресурсам пользователя без его пароля.

### Пример: «Войти через Google»

```
1. Пользователь нажимает "Войти через Google" на shop.com
2. Редирект на Google: "Shop.com хочет получить ваш email и имя"
3. Пользователь соглашается
4. Google перенаправляет обратно на shop.com с authorization code
5. Shop.com обменивает code на access_token у Google (server-to-server)
6. Shop.com запрашивает данные пользователя у Google API
7. Shop.com создаёт/находит пользователя, авторизует
```

### Роли в OAuth

| Роль | Кто |
|------|-----|
| **Resource Owner** | Пользователь (владелец данных) |
| **Client** | Приложение (shop.com) |
| **Authorization Server** | Выдаёт токены (Google, VK) |
| **Resource Server** | Хранит данные (Google API) |

## API Key

Простейший способ идентификации клиента. Ключ передаётся в заголовке или query-параметре.

```
GET /api/weather?api_key=abc123def456
# или
X-API-Key: abc123def456
```

| Плюсы | Минусы |
|-------|--------|
| Очень просто | Нет привязки к пользователю |
| Подходит для публичных API | Легко утечь (в URL, логах, коде) |

## Сравнение

| Метод | Stateful | Безопасность | Масштабируемость | Применение |
|-------|:---:|:---:|:---:|------------|
| Basic Auth | - | Низкая | Хорошо | Внутренние API |
| Session | Да | Средняя | Плохо | Монолиты, веб-приложения |
| JWT | Нет | Средняя | Отлично | SPA, мобильные, микросервисы |
| OAuth 2.0 | Нет | Высокая | Отлично | «Войти через...», доступ к API |
| API Key | Нет | Низкая | Отлично | Публичные API, интеграции |

## Что тестировщику проверять

- Запросы без токена → 401
- Запросы с чужим токеном → 403
- Истёкший access token → 401, refresh должен выдать новый
- Логаут инвалидирует токен/сессию
- Токен не передаётся в URL (только заголовки)
- JWT payload не содержит чувствительных данных (пароли, номера карт)
