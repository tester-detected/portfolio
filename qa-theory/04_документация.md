# Документация в тестировании

Тестовая документация — это артефакты, которые создаются до, во время и после тестирования. Она помогает планировать, выполнять и отслеживать результаты.

## Основные виды документации

| Документ | Зачем | Кто создаёт |
|----------|-------|-------------|
| Тест-план | Стратегия, сроки, ресурсы, scope | QA Lead / QA Manager |
| Тест-кейс | Пошаговый сценарий проверки | QA Engineer |
| Чек-лист | Краткий список проверок без деталей | QA Engineer |
| Баг-репорт | Описание найденного дефекта | QA Engineer |
| Отчёт о тестировании | Итоги прогона, метрики, риски | QA Lead |

## Тест-план

Документ верхнего уровня, описывающий **что**, **как**, **когда** и **кем** будет тестироваться.

### Структура тест-плана

1. **Введение** — цели тестирования, описание продукта
2. **Scope (Объём)** — что тестируем и что НЕ тестируем
3. **Подход к тестированию** — виды тестирования (функциональное, регрессионное, нагрузочное и т.д.)
4. **Критерии входа/выхода**
   - Entry criteria: сборка развёрнута, smoke пройден, тестовые данные готовы
   - Exit criteria: все критические баги исправлены, тестовое покрытие > 80%
5. **Ресурсы** — кто тестирует, какие инструменты, окружения
6. **Расписание** — этапы, дедлайны
7. **Риски** — что может пойти не так и как митигировать
8. **Deliverables** — какие артефакты будут предоставлены (отчёты, логи)

### Пример тест-плана (сокращённый)

```
Проект: Интернет-магазин v2.0
Дата: 02.04.2026
Автор: Патракеева Т.

1. Scope
   Тестируем: Корзина, Оформление заказа, Оплата, Личный кабинет
   Не тестируем: Админ-панель (отдельный цикл), SEO

2. Подход
   - Функциональное (ручное + автотесты Cypress)
   - Регрессионное (после каждого спринта)
   - API-тестирование (Postman)
   - Кроссбраузерное (Chrome, Firefox, Safari)

3. Критерии входа
   - Сборка развёрнута на staging
   - Smoke-тесты пройдены
   - Тестовые данные подготовлены

4. Критерии выхода
   - 0 блокеров и критических багов
   - Покрытие требований >= 90%
   - Регресс пройден

5. Риски
   - Нестабильная тестовая среда → запасной стенд
   - Изменения требований в последний момент → буфер 20% времени
```

## Тест-кейс

Детальный сценарий проверки конкретной функциональности с чёткими шагами и ожидаемым результатом.

### Структура тест-кейса

| Поле | Описание |
|------|----------|
| **ID** | Уникальный идентификатор (TC-001) |
| **Название** | Краткое описание, что проверяем |
| **Приоритет** | Critical / High / Medium / Low |
| **Предусловия** | Что должно быть выполнено до начала теста |
| **Шаги** | Нумерованный список действий |
| **Ожидаемый результат** | Что должно произойти на каждом шаге или в конце |
| **Тестовые данные** | Конкретные значения для ввода |

### Пример тест-кейса

```
ID: TC-042
Название: Успешное оформление заказа с оплатой картой
Приоритет: Critical
Модуль: Оформление заказа

Предусловия:
- Пользователь авторизован
- В корзине минимум 1 товар
- Товар в наличии на складе

Шаги:
1. Перейти в корзину
2. Нажать "Оформить заказ"
3. Заполнить адрес доставки: г. Москва, ул. Тестовая, д. 1
4. Выбрать способ оплаты: "Банковская карта"
5. Ввести данные карты: 4242 4242 4242 4242, 12/28, CVC 123
6. Нажать "Оплатить"

Ожидаемый результат:
- Отображается страница "Заказ оформлен"
- Номер заказа присвоен и отображается
- Статус заказа = "Оплачен"
- На email пришло подтверждение
- Остаток товара на складе уменьшился
- В личном кабинете заказ отображается в списке

Тестовые данные:
- Пользователь: testuser@example.com / Passw0rd
- Карта: 4242 4242 4242 4242 (тестовая)
```

## Чек-лист

Упрощённый формат: список проверок **без детальных шагов**. Быстрее создавать, удобнее для опытных тестировщиков.

### Когда использовать чек-лист вместо тест-кейса

| Чек-лист | Тест-кейс |
|----------|-----------|
| Опытная команда | Новые сотрудники в проекте |
| Исследовательское тестирование | Приёмочное тестирование |
| Частые изменения требований | Стабильные требования |
| Быстрые проверки | Детальная документация нужна заказчику |

### Пример чек-листа: Регистрация

```
Модуль: Регистрация пользователя
Дата: 02.04.2026

[✓] Регистрация с валидными данными
[✓] Регистрация с уже существующим email
[✓] Регистрация с пустым email
[✓] Регистрация с невалидным email (без @)
[✓] Пароль менее 8 символов
[✓] Пароль без заглавной буквы
[✓] Пароль без цифры
[✓] Повторный пароль не совпадает
[ ] Регистрация с кириллицей в имени
[ ] Email — регистронезависимость
[ ] SQL-инъекция в поле имени
[ ] XSS в поле имени
[ ] Отправка формы по Enter
[ ] Ссылка "Уже есть аккаунт?" ведёт на логин
```

## Баг-репорт

Описание найденного дефекта. Чем точнее описан баг — тем быстрее его исправят.

### Структура баг-репорта

| Поле | Описание |
|------|----------|
| **ID** | Автоматически из баг-трекера (JIRA-1234) |
| **Название** | Короткое, информативное: Что? Где? Когда? |
| **Severity** | Blocker / Critical / Major / Minor / Trivial |
| **Priority** | High / Medium / Low |
| **Окружение** | Браузер, ОС, стенд, версия приложения |
| **Предусловия** | Начальное состояние |
| **Шаги воспроизведения** | Нумерованный список |
| **Фактический результат** | Что происходит (баг) |
| **Ожидаемый результат** | Что должно происходить |
| **Вложения** | Скриншот, видео, логи |

### Severity vs Priority

| | Severity (тяжесть) | Priority (приоритет) |
|---|---|---|
| Определяет | Насколько баг влияет на систему | Насколько срочно чинить |
| Кто ставит | QA | PM / QA Lead |
| Пример: высокая S, низкий P | Баг в редко используемой функции, но она падает | Критичный для системы, но затрагивает 0.1% пользователей |
| Пример: низкая S, высокий P | Опечатка на главной странице | Не ломает ничего, но заметна всем |

### Пример баг-репорта

```
ID: BUG-789
Название: Корзина: при добавлении 0 штук товара сумма становится NaN

Severity: Major
Priority: High
Окружение: Chrome 120, Windows 11, staging.shop.example.com v2.0.3

Предусловия:
- Пользователь авторизован
- Каталог содержит товары

Шаги воспроизведения:
1. Открыть каталог товаров
2. Нажать "В корзину" на любом товаре
3. Перейти в корзину
4. Изменить количество товара на 0
5. Посмотреть итоговую сумму

Фактический результат:
Итоговая сумма отображается как "NaN руб."
Кнопка "Оформить заказ" активна

Ожидаемый результат:
- Количество 0 не допускается (минимум 1)
- ИЛИ товар удаляется из корзины при количестве 0

Вложения: screenshot_nan_cart.png
```

## Отчёт о тестировании

Итоговый документ после тестового цикла. Показывает готовность продукта к релизу.

### Структура

1. **Общая информация** — проект, версия, период тестирования
2. **Scope** — что тестировалось
3. **Статистика**
   - Всего тест-кейсов: 250
   - Пройдено: 230 (92%)
   - Провалено: 15 (6%)
   - Заблокировано: 5 (2%)
4. **Статистика дефектов**
   - Найдено: 42
   - Исправлено: 38
   - Открытых: 4 (0 Blocker, 1 Major, 3 Minor)
5. **Покрытие требований** — % требований, покрытых тестами
6. **Риски и рекомендации** — можно ли релизить, с какими оговорками
7. **Вывод** — GO / NO-GO решение

### Пример вывода

```
Рекомендация: GO с оговоркой

Все критические и блокирующие дефекты исправлены.
Открыт 1 Major-дефект (BUG-789: NaN в корзине) — 
workaround существует (валидация на фронте), 
фикс запланирован в hotfix v2.0.4.

Регрессионное тестирование пройдено: 92% pass rate.
Автотесты Cypress: 148/155 passed, 7 flaky (перезапущены успешно).
```
