# API-тестирование (REST, SOAP, WebSocket)

## Основы API

**API (Application Programming Interface)** — описание способов и правил, по которым одна программа может взаимодействовать с другой.

### Внешние и внутренние API

- **Внешние (сторонние) API** — доступны всем, включая сторонних разработчиков. Примеры: Google Maps, авторизация через соцсети.
- **Внутренние API** — разрабатываются под нужды конкретного сайта. Пример: поиск с фильтрацией.

### Тестирование API

Тестирование API сосредоточено на уровне бизнес-логики архитектуры ПО. Цель — проверка функциональности, надёжности, производительности и безопасности программных интерфейсов. Относится к интеграционному тестированию.

## REST vs SOAP vs WebSocket

| | REST | SOAP | WebSocket |
|---|---|---|---|
| **Формат** | JSON (чаще всего) | XML | Фреймы (любой формат) |
| **Протокол** | HTTP | HTTP, SMTP и др. | WS/WSS |
| **Модель** | Запрос → Ответ (stateless) | Запрос → Ответ (stateful возможен) | Постоянное двустороннее соединение |
| **Контракт** | Swagger/OpenAPI | WSDL (строгий) | Нет стандарта |
| **Применение** | Веб, мобильные, CRUD | Корпоративные системы, банки | Чаты, уведомления, real-time |

## HTTP-методы

- **GET** — получить данные, без изменения состояния
- **POST** — создать ресурс или выполнить действие
- **PUT** — полное обновление ресурса
- **PATCH** — частичное обновление
- **DELETE** — удаление ресурса

Основные коды: 200/201 — успех, 400 — ошибка клиента, 401/403 — проблемы с авторизацией/правами, 404 — не найдено, 500+ — ошибки сервера.

## Как тестировать REST API

- Проверяю позитивные и негативные сценарии, обязательность полей, валидацию, работу прав доступа
- Смотрю тело ответа, статус-коды, заголовки, время ответа
- Валидирую схему ответа (структуру JSON), сравниваю с документацией
- При интеграционных проверках просматриваю запросы через Charles/Proxyman/Fiddler

## Как тестировать WebSocket

- Проверяю установку соединения, авторизацию/токены
- Отправку и приём сообщений, корректность формата и реакцию на некорректные данные
- Поведение при потере соединения и переподключении

## Как тестировать SOAP

- Настраиваю endpoint и WSDL в SoapUI
- Проверяю корректность XML-запросов/ответов, обязательные поля
- Коды ошибок и бизнес-ошибки в теле ответа

## Инструменты

- **REST:** Postman, Insomnia, Swagger, curl
- **SOAP:** SoapUI
- **WebSocket:** Postman, wscat
- **Снифферы:** Charles, Proxyman, Fiddler
- **TMS-интеграция:** результаты связываются с тест-кейсами в TestRail, Allure TestOps, Testy
