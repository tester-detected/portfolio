# Виды тестирования

## По целям

### Функциональное
Проверка соответствия требованиям (работа кнопок, форм, бизнес-логики).
- **Дымовое (Smoke)** — базовая проверка: «приложение запускается и основные функции работают»
- **Регрессионное** — проверка, что изменения не сломали старое
- **Приемочное (Acceptance)** — финальная проверка заказчиком на соответствие требованиям
- **Интеграционное** — проверка взаимодействия между модулями/сервисами

### Нефункциональное
Проверка качества ПО (не «что делает», а «как делает»).
- **Производительность** — работоспособность, стабильность, потребление ресурсов
- **Нагрузочное (Performance/Load)** — работа системы при высокой нагрузке
- **Стрессовое (Stress)** — работа за пределами максимальной нагрузки
- **Совместимость** — работа в разных браузерах, ОС, устройствах
- **Надёжность** — способность работать в заданных условиях длительное время
- **Usability (Удобство)** — оценка интерфейса для пользователя
- **Security (Безопасность)** — поиск уязвимостей
- **Локализация** — адаптация под регион (язык, валюта, форматы дат)
- **Доступность (Accessibility)** — пригодность для людей с нарушениями слуха, зрения

## По уровням (пирамида тестирования)

```
        /\
       /  \        E2E (UI-тесты)         — медленные, дорогие, хрупкие
      /    \
     /──────\      Интеграционные          — средняя скорость и стоимость
    /        \
   /──────────\    Модульные (Unit)        — быстрые, дешёвые, стабильные
  /____________\
```

- **Модульное (Unit)** — тестирование отдельных функций/методов
- **Интеграционное** — тестирование взаимодействия модулей
- **Системное** — тестирование всей системы целиком
- **Приемочное** — тестирование заказчиком/пользователем

---

## По степени автоматизации

- **Ручное** — тестировщик выполняет тесты вручную
- **Автоматизированное** — тесты выполняются скриптами (Cypress, Playwright, Selenium)

---

## По знанию системы

- **Чёрный ящик** — без знания внутренней структуры. Тестируем только вход → выход.
- **Белый ящик** — полный доступ к исходному коду. Тестируем внутреннюю логику.
- **Серый ящик** — есть доступ к БД или фрагментам кода, но без углубления в детали реализации.

---

## По степени важности тестируемых функций

- **Дымовое (Smoke)** — самый поверхностный, но критичный. Проверяет: запускается ли приложение, работает ли авторизация. Если не пройдено — дальнейшее тестирование бессмысленно.
- **Критического пути (Critical Path)** — проверка наиболее часто используемых сценариев и основного функционала.
- **Расширенное (Extended)** — глубокое тестирование всей функциональности, включая редко используемые функции.

---

## По времени проведения и связи с изменениями

### Smoke vs Sanity

| | Smoke | Sanity |
|---|---|---|
| **Цель** | Сборка «жива», основные функции работают | Конкретные исправления работают корректно |
| **Объём** | Весь функционал поверхностно | Узкий функционал детально |
| **Когда** | С каждым новым релизом, часто автоматически | После багфиксов или при нехватке времени на полную регрессию |
| **Документирование** | Обычно есть скрипты | Часто без документации |

- **Регрессионное (Regression)** — проверка, что изменения не сломали старое
- **Альфа-тестирование** — внутреннее, до релиза
- **Бета-тестирование** — внешнее, с реальными пользователями
- **Приемочное (Acceptance)** — заказчик проверяет соответствие требованиям

---

## По принципам работы с приложением

- **Позитивное** — проверка корректной работы с валидными данными
- **Негативное** — проверка обработки невалидных данных, ошибок, граничных случаев
