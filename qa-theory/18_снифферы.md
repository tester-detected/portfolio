# Charles, снифферы и перехват трафика

## Что такое сниффер

**Сниффер (HTTP-прокси)** — инструмент, который перехватывает и показывает HTTP/HTTPS трафик между клиентом и сервером. Позволяет видеть, что реально отправляется и приходит.

## Зачем тестировщику

- Увидеть реальные запросы и ответы (не то, что показывает UI)
- Проверить, какие данные отправляет приложение (нет ли лишних)
- Найти ошибки, которые UI проглатывает молча
- Подменить ответ сервера для тестирования edge-cases
- Замедлить соединение для тестирования на слабых сетях
- Тестировать мобильные приложения (где нет DevTools)

## Популярные снифферы

| Инструмент | Платформа | Цена | Особенности |
|-----------|-----------|------|-------------|
| **Charles Proxy** | Windows, macOS, Linux | Платный (~$50) | Самый популярный, удобный GUI |
| **Proxyman** | macOS | Бесплатный / Pro | Современный UI, только macOS |
| **Fiddler** | Windows (Classic), все (Everywhere) | Бесплатный / Платный | Мощный, скрипты на C# |
| **mitmproxy** | Все | Бесплатный (open source) | CLI, Python-скрипты |

## DevTools vs Сниффер

| | DevTools (Network) | Сниффер (Charles) |
|---|---|---|
| Где работает | Только в браузере | Любое приложение (мобильное, десктоп) |
| HTTPS | Видит расшифрованным | Нужен корневой сертификат |
| Подмена ответов | Ограниченно | Полноценно (Map Remote, Map Local, Rewrite) |
| Throttling | Простой | Гибкий (по домену, по типу) |
| Breakpoints | Нет | Да (остановить запрос, изменить, отправить) |

## Charles Proxy — основные функции

### 1. Просмотр трафика

Видим все запросы: URL, метод, статус, заголовки, тело запроса и ответа. Можно фильтровать по домену.

### 2. SSL Proxying (HTTPS)

По умолчанию HTTPS-трафик зашифрован. Чтобы видеть его:
1. Proxy → SSL Proxying Settings → добавить домен (`*.example.com`)
2. Help → SSL Proxying → Install Charles Root Certificate
3. Установить сертификат как доверенный

**На мобильном:**
1. Настроить прокси на телефоне (IP компьютера + порт 8888)
2. Открыть `chls.pro/ssl` в браузере телефона
3. Установить сертификат

### 3. Map Local / Map Remote

**Map Local** — подменить ответ сервера файлом с диска:
- Проверить, как UI отображает пустой список, ошибку, огромные данные
- Не нужно менять бэкенд

**Map Remote** — перенаправить запрос на другой URL:
- Например, вместо prod API использовать staging

### 4. Rewrite

Изменение заголовков, тела запроса/ответа на лету:
- Добавить/удалить заголовок
- Изменить статус-код (200 → 500)
- Заменить текст в теле ответа

### 5. Breakpoints

Остановить запрос или ответ, отредактировать вручную и отправить дальше:
- Изменить тело запроса перед отправкой
- Изменить ответ сервера перед показом клиенту
- Полезно для тестирования валидации на сервере

### 6. Throttling (Замедление сети)

Имитация медленного интернета:
- 3G, Edge, 56 kbps
- Проверка таймаутов, загрузки, UX на слабой сети

### 7. Repeat / Compose

- **Repeat** — отправить запрос повторно (проверка идемпотентности)
- **Compose** — создать произвольный запрос с нуля

## Практические сценарии

### Проверить, что приложение не отправляет лишние данные
1. Включить запись трафика
2. Выполнить действие (логин, отправка формы)
3. Посмотреть тело запроса — нет ли паролей в открытом виде, лишних полей

### Проверить обработку ошибок сервера
1. Breakpoint на ответ от API
2. Изменить статус на 500, тело на `{"error": "Internal Server Error"}`
3. Проверить, показывает ли UI адекватное сообщение

### Проверить пустые данные
1. Map Local: подставить `{"items": [], "total": 0}`
2. Проверить, что UI показывает «Ничего не найдено», а не падает

### Тестировать мобильное приложение
1. Компьютер и телефон в одной сети
2. На телефоне: Wi-Fi → Proxy → IP компьютера:8888
3. Установить SSL-сертификат Charles
4. Весь трафик приложения виден в Charles
