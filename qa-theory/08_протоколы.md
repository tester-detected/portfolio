# Сетевые протоколы

## Модель TCP/IP

```
┌─────────────────────────────┐
│  4. Прикладной (Application) │  HTTP, HTTPS, FTP, SMTP, DNS, WebSocket
├─────────────────────────────┤
│  3. Транспортный (Transport)  │  TCP, UDP
├─────────────────────────────┤
│  2. Сетевой (Internet)        │  IP, ICMP
├─────────────────────────────┤
│  1. Канальный (Link)          │  Ethernet, Wi-Fi
└─────────────────────────────┘
```

## TCP vs UDP

| | TCP | UDP |
|---|---|---|
| Соединение | Устанавливает (3-way handshake) | Не устанавливает |
| Надёжность | Гарантирует доставку, порядок | Не гарантирует |
| Скорость | Медленнее (накладные расходы) | Быстрее |
| Применение | HTTP, email, файлы | Видео-стриминг, VoIP, DNS, онлайн-игры |

### TCP 3-way handshake

```
Клиент → SYN        → Сервер     "Хочу соединиться"
Клиент ← SYN-ACK    ← Сервер     "Ок, готов"
Клиент → ACK        → Сервер     "Подтверждаю, начинаем"
```

## TLS/SSL

**TLS (Transport Layer Security)** — криптографический протокол, обеспечивающий безопасную передачу данных. SSL — устаревшее название (SSL 3.0 → TLS 1.0).

### Что даёт TLS

- **Шифрование** — данные нельзя прочитать при перехвате
- **Аутентификация** — подтверждение, что сервер тот, за кого себя выдаёт (сертификат)
- **Целостность** — данные не изменены в пути

### Как работает TLS-handshake (упрощённо)

```
1. Client Hello     → Клиент отправляет поддерживаемые шифры, версию TLS
2. Server Hello     ← Сервер выбирает шифр, отправляет сертификат
3. Проверка          Клиент проверяет сертификат через CA (Certificate Authority)
4. Обмен ключами    → Генерация общего симметричного ключа
5. Шифрованное       Все данные шифруются общим ключом
   соединение
```

### HTTP vs HTTPS

| | HTTP | HTTPS |
|---|---|---|
| Порт | 80 | 443 |
| Шифрование | Нет | TLS |
| Данные | Видны при перехвате | Зашифрованы |
| Сертификат | Не нужен | Нужен (Let's Encrypt — бесплатно) |

## WebSocket (WS / WSS)

**WebSocket** — протокол полнодуплексной связи поверх TCP. В отличие от HTTP (запрос-ответ), WebSocket держит постоянное соединение.

### HTTP vs WebSocket

| | HTTP | WebSocket |
|---|---|---|
| Модель | Запрос → Ответ | Постоянное двустороннее соединение |
| Инициатор | Всегда клиент | И клиент, и сервер |
| Overhead | Заголовки в каждом запросе | Заголовки только при handshake |
| Применение | REST API, загрузка страниц | Чаты, уведомления, real-time данные |

### WS vs WSS

- `ws://` — без шифрования (как HTTP)
- `wss://` — с TLS-шифрованием (как HTTPS)

### Handshake

```
Клиент → GET /chat HTTP/1.1
         Upgrade: websocket
         Connection: Upgrade

Сервер ← HTTP/1.1 101 Switching Protocols
         Upgrade: websocket

→ Соединение установлено, обмен фреймами
```

## FTP (File Transfer Protocol)

Протокол передачи файлов. Работает поверх TCP, порты 20 (данные) и 21 (управление).

| | FTP | SFTP | FTPS |
|---|---|---|---|
| Шифрование | Нет | SSH | TLS |
| Порт | 21 | 22 | 990 |
| Безопасность | Пароли в открытом виде | Безопасно | Безопасно |

**Для тестировщика:** FTP используется для загрузки/выгрузки файлов на сервер. Проверяем: корректность загрузки, права доступа, размер файлов, кодировку имён.

## SMTP (Simple Mail Transfer Protocol)

Протокол отправки электронной почты. Порт 25 (без шифрования), 587 (с STARTTLS), 465 (с SSL).

```
Отправка:  Клиент → SMTP-сервер → SMTP-сервер получателя
Получение: POP3 (скачивает и удаляет) / IMAP (синхронизирует)
```

**Для тестировщика:** Проверяем отправку email (регистрация, восстановление пароля, уведомления). Инструменты: Mailtrap, MailHog — ловят письма без реальной отправки.

## DNS (Domain Name System)

Преобразует доменное имя в IP-адрес.

```
Браузер: "Мне нужен example.com"
  → Кэш браузера → Кэш ОС → DNS-резолвер провайдера
  → Root DNS → .com DNS → example.com DNS
  ← 93.184.216.34
```

**Для тестировщика:**
- DNS-кэш может вернуть старый IP после миграции → `ipconfig /flushdns`
- TTL (Time to Live) определяет, как долго кэшировать запись
- Проверка: `nslookup example.com`, `dig example.com`
